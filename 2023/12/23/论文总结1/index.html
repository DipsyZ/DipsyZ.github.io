Accelerating Large-Scale Prioritized Graph Computations by Hotness Balanced Partition

IEEE TRANSACTIONS ON PARALLEL AND DISTRIBUTED SYSTEMS, VOL. 32, NO. 4, APRIL 2021

摘要：

优先计算在许多图算法中表现出良好的性能。优先计算优先执行那些在指定收敛中起重要作用的顶点。对于大规模分布式图计算来说，图分区是一个重要的预处理步骤，目的是负载均衡和降低节点之间的通信成本。
然而，现有的图分区方法是为轮询round-robin同步分布式框架设计的。他们在不区分顶点重要性的情况下负载均衡，并且没有考虑基于优先级调度的特性，这可能会限制优先图计算的好处。
在本文中，为了加速优先迭代图计算，我们提出了 Hotness Balanced Partition (HBP)。在优先图计算中，高优先级的顶点可能会被更频繁地执行，并且可能会传递更多的消息，从而导致热点顶点hot vertices。基于这一点，文章通过根据热度有区别地分配顶点而不是盲目地分配权重相等的顶点来划分图，目的是将hot vertices均匀地分配给workers。
文章进一步提供了两种 HBP 算法：a streaming-based algorithm for efficient one-pass processing和a distributed algorithm for distributed processing。我们的结果表明，我们提出的分区方法优于最先进的分区方法 Fennel、HotGraph 和 SNE。

1.Introduction

分布式图处理系统： 将大规模图分片，每一个集群上的worker负责处理一个partition:

  1. 大量的 点/边分割（edge/vertex cut）会加大通信成本。
  2. 由于各节点工作负载不均衡可能会导致出现闲置的worker，降低计算效率。

Prior graph partitioning algorithms：都采用了一个假设：底层的分布式图框架采用synchronous parallel processing model，在每一次迭代(super step)之后有一个全局同步屏障 barrier

在 round-robin 机制下，实现简单而使用广泛。在每次迭代中，每个点被执行一次，每条边只传递一条信息。因此，在图分区中，点的数量（vertex-centric frameworks）/边的数量（edge-centric frameworks）表明了worker的工作量，the number of cutting edges/ vertex replicas in vertex-cut systems表明了通信的成本。->故：现存的图分割算法的目标：1.平衡点/边数量；2.减少切边/点分割的数量
asynchronous parallel processing：近期采用的一些图系统： GraphLab [2],GraphUC [12], Giraph++ [13], GRAPE+ [14] and Maiter。消除了全局同步障碍，worker之间不必等待，边/点可以在任何时间执行，取消了一次迭代只能执行一次的约束。所以相同时间间隔内有些点/边比其他执行的次数更多，在近期的研究中发现有些点对于最终收敛结果更加重要。在异步计算中，顶点的重要性不是持续不变的。在优先计算中，只有一部分顶点在一轮迭代中执行，这些顶点计算的更加频繁。

优先计算比轮询计算平均快1.2倍。

Example 1. prioritized PageRank算法采用priority scheduling（优先调度），每次迭代执行顶点子集。传统的图分割会限制prioritized PageRank计算性能

图划分：
通用算法：哈希划分算法。为每个数据项(点或 者边)计算一个哈希值，将其映射到一个划分出的子图中

图流划分streaming graph partition, SGP：是图划分问题的一个新的衍生。与传统图划分算法相比，图流划分算法不在执行划分算法的计算机上存储完整的图，而是对持续不断地读入的边或者点进行实时划分。这些读入的点或边可以源自计算机集群的硬盘，或者是直接从数据源获取的动态数据。由于不需要在单机上存储完整的图数据，图流划分算法相比传统图划分算法，可以应用于更大规模的图，同时具有时间复杂度低，可以处理动态数据的优势。但是由于其缺少完整的图数据，图流划分的难度更大，而且划分效果往往略差于传统图划分算法。
图流划分两种：

  1. 点划分使用点流（vertex stream）模型，每次读入的数据项为一个点及其完整的邻接列表。
  2. 边划分使用边流（edge stream）模型，每次读入的数据项为一条单独的边。

Fennel:2014年WSDM论文《Fennel: Streaming graph partitioning for massive scale graphs》,将图流划分问题划归为流上的模块最大化 (modularity maximization) 问题.
Minimizing the number of cut edges and balancing of the partition sizes.
Place a vertex to the partition with the least number of non-neighbors

Fennel runtime更高因为没有考虑到点的优先级。如下图

a. high-priority点需要更高的计算资源，节点间负载不均衡，会导致计算收敛慢。
b. 对于high-priority点之间edge-cut会导致，通信成本剧增，runtime也会上升。

Motivation：

  1. workload balance并不意味着着计算高效。prioritized processing消除了synchronous barriers,，不同机器之间不用彼此等待，没有空闲节点。对收敛重要的顶点需要更多计算资源，在图分割中需要重点考虑如何处理这些顶点。
  2. 小数量的边切割/顶点复制不一定意味较少的通信资源开销。
    —>需要一个针对prioritized processing的图分割方法

Contribution: Hotness Balanced Partition (HBP)

  1. SPb-HBP(Streaming-based Per-Bin Hotness Balanced Partition)启发式的基于图流模型的图分割
  2. DSPb-HBP分布式场景

5 DISTRIBUTED SPB-HBP

前提：大图数据——>分布式存储——>获得neighborhood vertex信息需要多轮同步——>网络拥堵，计算表现不佳

解决方案：

  1. 每个worker将自己拥有的vertex/edge信息压缩成一个local graph summary(包含：topology characteristics、vertex/edge hotness),由SPb-HBP完成。
  2. local graph summary 提交给 global worker，分发各worker，之后解压。
  3. 只有 graph summary 在 worker 之间传递，降低通信开销。


1.local compression

super-vertex principles:

  1. The vertices that have higher communication cost should be combined in a super-vertex, in order to avoid cross-partition communication.
  2. The vertices that have higher communication cost should be combined in a super-vertex, in order to avoid cross-partition communication.
  3. The vertices that have the same neighbor should be combined in a super-vertex, in order to reduce the number of super-edges.
  4. Only the vertices that are in the same hotness bin can be combined, in order to keep the hotness bin information for the next global partition phase.
